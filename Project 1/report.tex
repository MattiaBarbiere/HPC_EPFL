\section{Assessing inefficiencies: Profiling}
The first step to any parallelization is understanding the bottleneck in the code. After taking some time to understand the code I performed a sequential profiling with \texttt{perf} on the unaltered code. Please note that throughout this report the code was compiled without any optimizer so that results are more evident.\\
From \Cref{fig:perf_before_para} it is clear that the main bottleneck that we can control is the \texttt{mat\_vec} function that performs matrix vector multiplication on the sparse matrix. By simple inspection the \texttt{CGSolverSpare::solve} function, that implements the conjugate method, is inherently sequential and thus cannot be parallelized.

\section{Dividing the work: MPI parallelization}
After having found the candidate for parallelization the next step was the implementation using MPI. The crux of the \texttt{mat\_vec} function is the \texttt{for} loop that performs the multiplication and addition of the matrix and vector. The strategy I invoked to parallelize this function was to evenly split the non-zero indices of the matrix among the given processors. The processors would then compute the additions and multiplication only on their portion of the matrix and have zero everywhere else. Lastly I performed a \texttt{MPI\_Allreduce} with the \texttt{MPI\_SUM} operation to combine all the results. Next I timed how long the \texttt{mat\_vec} function took to compute as I varied the number of processors. The speedup and efficiency of the \texttt{mat\_vec} function are presented in \Cref{fig:speed_eff_mat_vec}.

We clearly see that there is an initial speedup when we go from $1$ processor to $25$, however after this there is a evident plateau where the additional processors have no effect on the speed up. For the same reason the efficiency starts very close to one but eventually decays to $0$ as additional processors are of little help. This is the first indication that the code might not scale that well and in fact the same pattern emerges from weak and strong scaling.

\section{The bigger picture: Strong and weak scaling}
The parallelization of the \texttt{mat\_vec} function is only one part of the code. We need to look at the code in its entirety. To estimate the parallelizable code in Amdahl’s law I measured the proportion of time that the \texttt{mat\_vec} function was running compared to the total time of the whole code and I got $\alpha \approx 0.75$. Hence, according to Amdahl’s law, the ideal speedup should be $S(p) = \frac{1}{0.25 + \frac{0.75}{p}}$ and so should tend towards 4 as the number of processors $p$ goes towards infinity. Unfortunately, this is far from what happens in practice. In Figure \ref{fig:ss} we see that the empirical speedup starts close to the ideal value but then deviates very early with only a slight gain for the first few processors. In fact as the number of processes increases we actually get a slower performance as the speedup drops below 1. This is likely attributed to the memory and communication bottlenecks. With more processors there is more data to move around and moreover having more processors does not improve the speed of memory read and write.
Next to perform weak scaling and apply Gustafson’s law I needed to construct matrices of the correct size so that the work per processors is constant. I started from the \texttt{lap2D\_5pt\_n2000.mtx} and extracted the top left portion of various sizes. In the end I ended up with the amount of non-zero entries per process at about $k = 119780$. I then timed the code with $p$ processors keeping non-zero entries per processor constant and compared it to the performance of a single processor on the same matrix. The ratio gave me the values of $\alpha(N)$ show in Table \ref{tab:thin_table_13_columns}. Gustafson’s law then tells us that the speedup should be $1+ (p-1)\alpha(N)$ which is plotted in red on Figure \ref{fig:ws}. Similarly to before both the speedup and the efficiency start close to the ideal but also deviate quickly. We see the same pattern that there is an effective speed up in the code only for low values of processor counts. As soon as the number of processor goes above 20 the extra processors only add memory and communication complexity without any added benefit in speed. The reason behind this is again that the problem becomes memory bound and adding additional processors will not help, and actually ends up hurting the performance because of the additional complexity in communication.

\section{Conclusion and further considerations}
This report focuses on parallelizing the conjugate gradient method for solving linear systems, with an emphasis on the \verb|mat_vec| function. I evaluated its performance under both strong and weak scaling. Although the main effort was on parallelizing \verb|mat_vec|, further optimizations are possible. As the code is now memory-bound, allocating only the relevant matrix segments to each processor could enhance performance further.

Firstly, let's get an idea about the computational complexity of the code. For a single iteration of the CG method most operations take $O(N)$ where $N$ is the size of the vector $b$, however the matrix vector product takes $O(N_{nz})$ where $N_{nz}$ is the number of non-zero entries of the matrix. Thus, for $k$ iterations, the time complexity is $O(k(N_{nz} + N))$. A similar argument would show that the memory scales as $O(N_{nz} + 4N)$ to store $b, x_k, r_k$ and $p_k$. AS previously discussed only the matrix vector product can be parallelized, hence for $p$ processors we have a time complexity of $O(k(\frac{N_{nz}}{p} + N))$
